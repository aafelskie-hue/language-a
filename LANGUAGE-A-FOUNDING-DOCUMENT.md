# Language A — Founding Document

**A pattern language for contemporary living.**
Published by Beachhead Systems.

---

## What This Is

Language A is an original pattern language — a network of interconnected design principles for how we build neighborhoods, homes, and the details that make them work. Each pattern identifies a recurring design problem, presents the evidence, and proposes a spatial solution.

The tradition of pattern languages in architecture and design is well-established. Language A builds on that tradition with 21st-century evidence, cold-climate specificity, and an engineering practitioner's eye. It is written from Edmonton, Alberta — a city where winter means -30°C, where the growing season is 120 days, and where the question "does this building actually work?" is answered every January.

The patterns span three scales:

- **Neighborhoods** — how we organize daily living, mobility, and community
- **Buildings** — how homes, workplaces, and shared spaces are shaped
- **Construction** — materials, envelopes, details, and the making of things

---

## The Format

Each pattern follows a consistent structure:

- **Number and Name** — with a confidence rating (★★ high confidence, ★ moderate, ☆ speculative)
- **Upward connections** — larger-scale patterns that give this one context
- **Problem** — the recurring design problem, stated as a tension between forces
- **Evidence** — what we know from research, practice, and observation
- **Therefore** — the solution principle, stated as a spatial design move
- **Downward connections** — smaller-scale patterns that help implement this one

The confidence ratings are honest assessments, not marketing. A ★★ pattern has strong evidence and has been tested in multiple contexts. A ★ pattern has good reasoning but needs more testing. A ☆ pattern is a hypothesis worth exploring. These ratings can change as evidence accumulates.

---

## Pattern Status

Each pattern carries a status:

- **Published** — reviewed, refined, evidence verified
- **Candidate** — first draft, under editorial refinement
- **Proposed** — submitted, not yet reviewed

All patterns in the initial release are Candidate status. They graduate to Published through editorial review.

## Handling References to the Design Tradition

Language A exists within a rich tradition of architectural thought. Rather than pretending this tradition doesn't exist, we handle it honestly:

**Within pattern text:** References to prior thinkers and their ideas use scholarly attribution — "the pattern language tradition established by Christopher Alexander in 1977 proposed that..." or "building on Carlos Moreno's fifteen-minute city research..." This is normal academic practice.

**Within the connection network:** Patterns connect only to other Language A patterns. No external pattern numbers. Where a seed pattern currently references an external concept (e.g., "connects to THICK WALLS (197)"), we either:
1. **Write our own pattern** for that concept (→ becomes "Deep Walls" in Language A), or
2. **Use a generic description** of the principle (→ "...connects to principles of thermal mass and acoustic isolation...")

**The rule:** A reader should be able to understand and use Language A without having read any other book. The language is self-contained.

## The Platform

Language A is both a body of content and a digital platform for exploring it.

### Core Features
- **Pattern Explorer** — browse, search, filter by scale and category
- **Pattern Detail** — full content with connected patterns as navigable links
- **Network Graph** — interactive visualization showing patterns as nodes, connections as edges
- **Project Builder** — select patterns for a specific project, track status, add notes
- **AI Guide** — natural language interface for pattern recommendations

### Design Direction
The same "architectural editorial" aesthetic from the v1 prototype applies. Warm, literate, craft-forward. Not a software product — a reference work that happens to be interactive.

### Tech Stack
- Next.js 14+ (App Router), TypeScript, Tailwind CSS
- D3.js for network visualization
- Zustand for state management
- Static JSON for pattern data
- localStorage for projects
- Vercel for deployment

### Data Model

```typescript
interface Pattern {
  id: number;                     // Sequential from 1
  name: string;
  scale: "neighborhood" | "building" | "construction";
  category: string;               // e.g., "climate-resilience"
  confidence: 0 | 1 | 2;
  status: "published" | "candidate" | "proposed";
  problem: string;
  body: string;                   // Evidence/discussion (markdown)
  solution: string;               // The "therefore" statement
  connections_up: number[];       // Larger-scale pattern IDs
  connections_down: number[];     // Smaller-scale pattern IDs
  tags?: string[];                // e.g., ["cold-climate", "edmonton"]
  evidence_sources?: string[];    // Key citations
}
```

Note: `scale` uses "neighborhood" instead of "towns" — Language A's own vocabulary.

## What Language A Is Not

- It is not a revision of any existing book
- It is not a commentary on or companion to another work
- It is not a building code or technical standard
- It is not urbanism advocacy or policy recommendations

It is an original, interconnected set of design principles for building neighborhoods, homes, and details that support contemporary life — written by a structural engineer in a cold city, grounded in evidence, and designed to be useful.
